KUQuest MVP Database Design

Database-first design for Better Auth, marketplace work, disputes, wallets, immutable accounting, Xendit money movement, operations, and audit. This document explains both the tables and the guarantees they are intended to enforce.

1. Approved Design Decisions

The MVP uses one PostgreSQL database and the default public schema. Drizzle files are split by module, while cross-module transactions remain possible. A user is never forced into an employer/worker role: every ordinary user may do both.

Identity

Better Auth owns opaque text IDs for user, session, account, and verification. Domain records use PostgreSQL UUIDs and reference users with text foreign keys.

Authentication

Google login is accepted when the normalized address ends exactly with @ku.th. Better Auth's opaque database-backed session cookie authenticates API requests. There are no application JWTs.

Money units

Wallet and ledger columns use signed bigint whole baht and explicit _baht names. Exact provider fee, tax, and reconciliation values use signed bigint satang and _satang names. Currency is THB.

History

Main lifecycle rows keep current status for fast reads. Separate append-only history tables retain every transition with actor/source, reason, and time. Accounting history is never rewritten.

Provisioning

Creating a user automatically creates their wallet and required user ledger accounts in the same database transaction through a reviewed custom migration trigger.

MVP operations

One API process also runs small PostgreSQL-backed background loops. Compose has PostgreSQL, a one-shot migration service, and the API—no Redis and no separate worker.

Deliberate MVP exception: payout destination/profile fields are plaintext to make development debugging simple. They must be excluded from logs and API responses must mask them. Encryption at rest and key rotation are a mandatory production blocker.

Better Auth remains the owner of its four tables and token behavior, including the branch's encrypted OAuth token configuration. The API resolves the current user from the Better Auth session cookie. User IDs supplied by clients never determine ownership.

4. Jobs, Applications, Submissions, and Disputes

A published job is already fully funded. Job creation, the employer spending debit, and the job-held credit occur atomically. Current status is duplicated on the aggregate for fast queries; its history is the explanation and audit trail.

TableKey columnsPurpose and rules
jobsid uuid PK, employer/selected worker IDs, title/description, amount_baht, fee snapshot, policy revision, deadlines, current statusFunded listing and aggregate root. Positive whole-baht amount; employer cannot be selected worker. Selection deadline precedes work deadline. One selected worker at most. Status: OPEN, ASSIGNED, OVERDUE, IN_REVIEW, DISPUTED, SETTLED, RETURNED, CANCELLED, EXPIRED.
job_status_historyid uuid PK, job_id, from/to status, actor/source, reason, occurred_atAppend-only record of every transition. First row has no prior status. Source distinguishes user, administrator, system loop, and provider-derived action.
job_applicationsid uuid PK, job/worker IDs, message, current status, timestampsWorker request to be selected. Unique (job_id, worker_user_id). An employer cannot apply to their own job. Selection atomically accepts one application and withdraws/rejects other pending applications according to domain rules.
job_application_status_historyid uuid PK, application ID, from/to status, actor/source, reason, timeFull application lifecycle (PENDING, SELECTED, WITHDRAWN, REJECTED, REMOVED_WORKER_ENGAGED).
work_submissionsid uuid PK, unique job_id, worker ID, summary, status, submitted/review deadlineOne MVP submission per job. It starts the 24-hour review window. Evidence files are deferred; text summary remains available. Status history records submission/review actions.
work_submission_status_historyid uuid PK, submission ID, from/to status, actor/source, reason, timeAppend-only transitions such as SUBMITTED, APPROVED, AUTO_APPROVED, and DISPUTED.
disputesid uuid PK, unique job_id, opened-by user, reason code, description, current statusAt most one dispute per MVP job. Only a job party may open it in the allowed window. Status: OPEN, PENDING_SECOND_APPROVAL, RESOLVED. Time alone never resolves it.
dispute_status_historyid uuid PK, dispute ID, from/to status, actor/source, reason, timeAppend-only dispute progress. Operational escalation may be recorded without changing the financial outcome.
dispute_resolutionsid uuid PK, dispute ID, proposer ID, outcome, reason, amount/policy snapshots, statusBinary proposal: RETURN_TO_EMPLOYER or SETTLE_TO_WORKER; no partial award. Below threshold one authorized admin may execute. Above threshold it waits for a second administrator.
dispute_resolution_approvalsid uuid PK, unique resolution ID, approver ID, decision, reason, timeSecond-person control. Database check plus transaction logic require approver to differ from proposer. Execution references the resulting ledger transaction exactly once.

Critical marketplace indexes

  • jobs(status, application_deadline_at), jobs(status, work_deadline_at), and work_submissions(status, review_deadline_at) support the in-process scheduler.
  • A partial unique index on jobs(selected_worker_user_id) WHERE status IN ('ASSIGNED','OVERDUE','IN_REVIEW','DISPUTED') enforces one engaged job per worker.
  • job_applications(worker_user_id, status) supports atomic removal when a worker becomes engaged.
  • Status-history indexes use (aggregate_id, occurred_at DESC, id DESC) for deterministic timelines.

5. Wallet and Immutable Ledger

The ledger is the source of truth. wallets contains locked, transactionally maintained cached balances for fast reads and concurrency control. A rebuild must reproduce those caches from postings. “Held for jobs” and “reserved for payouts” are summaries of ledger accounts, not extra spendable wallets.

TableKey columnsPurpose and rules
walletsid uuid PK, unique user_id, cached spending/earnings baht, status, timestampsExactly one per user. Balances are non-negative bigint. ACTIVE/FROZEN governs new user commands but does not stop committed settlement or provider recovery work.
wallet_status_historyid uuid PK, wallet ID, from/to status, actor ID, reason, timeEvery freeze/unfreeze is retained. No hard deletion of financial history.
ledger_accountsid uuid PK, code, type, currency, optional wallet/user ownerEach wallet owns spending and earnings accounts. Central system accounts hold job commitments, payout clearing, provider assets, platform revenue, adjustments, and shortfalls; the transaction business reference identifies the job or payout. Valid owner/type combinations and account codes are constrained and unique.
ledger_transactionsid uuid PK, unique business reference, event type, idempotency key link, sealed_at, created metadataImmutable header for one business money event. Sealing requires at least two postings, one currency, and a zero sum; no posting may be appended afterward. A correction is a new compensating transaction referencing the original, never an update.
ledger_postingsid uuid PK, transaction/account IDs, amount_baht bigint, currencySigned non-zero posting. Sum of postings for each transaction and currency must be zero. Postings and completed headers reject update/delete through triggers.
idempotency_keysid uuid PK, user/scope/key, request hash, resource reference, response/status, expiryUnique (principal_user_id, operation_scope, key). Same payload returns the original result; a different hash conflicts. Provider webhook deduplication is separate.
earnings_conversionsid uuid PK, user ID, amount_baht, unique ledger transaction ID, timeImmutable command record for the fee-free one-way movement from earnings to spending. The balanced ledger transaction is the monetary source of truth.
wallet_activitiesid uuid PK, user ID, type/status, four bigint deltas, typed resource reference, timeUser-facing chronological projection of spending, earnings, job-held, and payout-reserved effects. It is written atomically and rebuildable; it is not raw ledger history.
wallet_adjustmentsid uuid PK, wallet/admin IDs, amount/compartment, reason, ledger transaction IDAdmin command record tied one-to-one to a compensating ledger transaction. There is no “set balance” operation.
amounts_owedid uuid PK, user ID, amount/recovered baht, reason/source, statusTracks value that cannot be recovered without making a wallet negative. It is operational debt, never a negative wallet balance.
flowchart LR C[Business command] -->|lock aggregate + wallet| T[Ledger transaction] T --> P1[Posting: source -N baht] T --> P2[Posting: destination +N baht] P1 --> B{Deferred balance check} P2 --> B B -->|sum = 0 and caches nonnegative| K[Commit] B -->|invalid| R[Rollback entire command]
Custom SQL required: a deferred constraint/finalization mechanism rejects zero-posting, one-posting, mixed-currency, unbalanced, or late-appended transactions; immutable triggers reject update/delete; wallet provisioning and cache maintenance are transactional. These guarantees cannot be represented by Drizzle TypeScript definitions alone.

Single-process loop behavior

The API starts bounded polling loops after readiness. Each loop claims a small batch, commits quickly, and uses database time. Tasks must be idempotent because the process can crash after applying an effect but before marking a task complete. A lease timeout makes abandoned work retryable. Multiple future API replicas can safely claim different rows through row locks, but this MVP has no independent worker autoscaling.

Known MVP limitation: heavy work can compete with HTTP traffic and API downtime pauses scheduled processing. The table design intentionally permits moving the same handlers into a separate worker later without a database redesign.

General indexing rules

  • Every foreign key used for joins or deletion checks receives a B-tree index; PostgreSQL does not create these automatically.
  • Queue indexes are partial and ordered, for example (run_at, id) WHERE status IN ('PENDING','RETRYABLE').
  • User history uses (user_id, created_at DESC, id DESC) for cursor pagination. Provider references and business references are unique.
  • Use partial indexes for “active/current/non-terminal” uniqueness rather than mutable boolean-only conventions.
  • Use check constraints for local row facts; use locked service transactions or custom triggers for cross-row/state-machine facts.

9. Security Boundaries

One database improves atomicity, so least privilege must be explicit. Network isolation and separate credentials prevent the API from turning an injection or application compromise into schema ownership.

PrincipalMay doMust not do
kuquest_migratorOwn schema objects; run reviewed DDL, migrations, grants, functions, and triggers.Serve HTTP requests or share credentials with the runtime.
kuquest_appConnect at runtime with table-specific rights: Better Auth may delete ephemeral session/account/verification rows; user and domain history are retained; ledger/history writes use only narrowly permitted inserts or protected functions.DDL, role management, superuser, table ownership, TRUNCATE, trigger control, broad mutation of cached balances, or bypass of immutable-ledger protections.
PostgreSQL bootstrap/adminCreate the two roles/database during local initialization or controlled infrastructure setup.Be exposed to the API container or used as an application connection string.
  • Revoke default CREATE on public and unnecessary PUBLIC privileges. Grant existing objects explicitly. Before production, add a reviewed default-privilege policy so future objects remain inaccessible until deliberately exposed.
  • Trigger functions pin a safe search_path, qualify referenced objects, and are SECURITY INVOKER unless elevation is unavoidable. SECURITY DEFINER functions have EXECUTE revoked from PUBLIC; moving their ownership from the local login-capable migrator to a dedicated non-login function owner is a production requirement.
  • Keep PostgreSQL off public interfaces in production; Compose exposes it only when local debugging requires it.
  • Secrets come from environment/secret management, never the image or repository. Use separate migration and runtime URLs.
  • Better Auth cookies are HttpOnly, Secure in production, and appropriately SameSite; CORS/trusted origins are allowlisted.
  • Cookie-authenticated custom POST/PUT/PATCH/DELETE routes validate trusted Origin/Referer as CSRF defense. CORS alone is not CSRF protection, and Idempotency-Key is included in allowed request headers.
  • API authorization derives the user from the session. It never trusts a request body/path user_id as proof of ownership.
  • Webhook secrets are compared safely; raw payload access is restricted; logs and audits redact credentials, OAuth tokens, session tokens, and payout plaintext.

10. Drizzle Schema and Migration Policy

Drizzle TypeScript schemas are the structural source of truth and are split by module while exporting one combined schema to the client. Suggested files are auth.schema.ts, access.schema.ts, jobs.schema.ts, ledger.schema.ts, payments.schema.ts, and operations.schema.ts. Reviewed SQL migrations remain authoritative for grants, triggers, and invariants that Drizzle cannot express.

  1. Change the TypeScript schema, then create a versioned migration with drizzle-kit generate.
  2. Review generated SQL. Add a tracked custom migration—created with the Drizzle custom-migration workflow—for roles/grants, partial or deferred invariants, immutable triggers, wallet provisioning, and ledger/cache functions.
  3. Apply migrations with drizzle-kit migrate using kuquest_migrator. Never use drizzle-kit push on shared or production databases.
  4. Test both a clean database and upgrade from the previous migration. Include rollback/recovery instructions for non-transactional operations.
  5. Use only Drizzle's migration journal; do not retain the checkpoint's separate custom migration runner. Readiness probes the expected schema/journal version and fails when migrations are absent. The API runtime never attempts DDL.

PostgreSQL bigint maps to JavaScript bigint when values may exceed the safe-number range. API serializers must convert deliberately because JSON cannot encode native bigint; never silently cast an unsafe amount to number.

Better Auth caution: preserve the adapter's expected table and column contract when changing auth. Generate and inspect its required schema changes rather than renaming library-owned columns casually.

12. Explicit Deferrals and Production Gates

Chat and media

Detailed chat and media-server schemas are deliberately out of scope for this implementation. Do not add file paths, storage URLs, blobs, or chat-specific attachment tables to the domain schema now. The future boundary is an opaque mediaId: submissions, disputes, avatars, portfolios, and chat messages will reference centralized media assets without knowing their physical storage location. A later design must add typed link tables and authorization/lifecycle rules.

Unspecified product features

Categories, skills, ratings, favorites, organizations, notifications, profiles, portfolios, and other marketplace discovery features have no approved requirements in the reviewed sources. They are intentionally absent rather than guessed into the schema.

Before live money

  • Replace development plaintext payout fields with authenticated encryption and a tested rotation/key-management design.
  • Confirm Xendit Thai collection/payout APIs, callbacks, routing fields, fee/tax behavior, late success, reversals, permissions, and one-merchant marketplace use.
  • Obtain legal/accounting review and define reconciliation ownership, alerting, shortfall response, and dispute staffing.
  • Pass concurrency, double-spend, idempotency, duplicate/missed webhook, task crash-retry, ledger rebuild, reversal, and negative-balance tests.
  • Threat-model database privileges, session/cookie/CORS behavior, admin separation of duties, raw provider payloads, logs, backups, and production network exposure.
Design status: the MVP schema, migrations, Better Auth session integration, wallet conversion slice, and Docker verification are implemented.
